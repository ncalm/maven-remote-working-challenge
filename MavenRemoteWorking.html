Maven Analytics - Remote Working Survey Challenge¶

About the dataset¶

This dataset contains the results of two surveys of 1,500 remote workers commissioned by the state of New South Wales (NSW). The first took place in August and September 2020, and aimed to capture the remote work experiences and attitudes at the height of the COVID lockdown. Once health restrictions were reduced and a return to lockdown seemed unlikely, a similar survey was conducted between March and April 2021.

To be eligible, survey respondents had to be employed NSW residents with remote working experience in their current job. This means that both unemployed people and those whose job cannot be done remotely were not taken into account. The resulting sample represents roughly 59% of the NSW workforce.

The responses are stored in 2 csv tables, one for each survey.

Each record represents a single respondent, and contains answers to questions on:

  • Their employers policies, practices, and attitudes for remote work
  • Their own attitude towards working remotely
  • The amount of time they spent working remotely, and how they spent it
  • The barriers they faced when working remotely, and how they have changed
  • The best and worst aspects of working remotely
  • Their expectations for working remotely in the future

The dataset can be downloaded here: https://maven-datasets.s3.amazonaws.com/Remote+Working+Survey/NSW+Remote+Working+Survey.zip

How to play the Maven Remote Work Challenge¶

For this challenge, your task is to assess the productivity and morale implications of working remotely and outline an ideal policy for the post-pandemic future, presented in the form of a single-page report or dashboard.

To make analysis easier, one goal will be to append both datasets together. This is not necessarily a trivial task since:

  • The surveys in each year may have asked different questions (question is in one year and not the other)
  • Questions which are asking the same thing may be asked in a different way, or rather, the column header is different for the same question
  • Questions asked in both years may present different but equivalent options (e.g. "Somewhat likely" vs. "Somewhat agree")

In order to use fuzzy matching more quickly than the lambda method above, we can use process.extract from fuzzywuzzy.

This method takes an item, then compares it to every item in a list.

So, in this example, we'll take each column header from one year and compare it with all the column headers from the other year.

To do this, we need the column headers as lists.

We can then put both lists in a list - a list of lists - then apply the same matching process in both directions (just in case).

In [10]:
# create a list of the two lists
cols = [col20,col21]

# then, for each of those lists
for ls in cols:
    
    # we want to match items in 2020 with the list for 2021
    # and match items in 2021 with the list for 2020
    if cols.index(ls) == 0:
        matches = cols[1]
    else:
        matches = cols[0]
    
    # for each item in the current list
    for i in ls:
        
        # where the col_name column is the current item and the year is the current year
        # cols.index(ls) = 0 for 2020, 1 for 2021
        mask = (df_colmatch.col_name == i) & (df_colmatch.year == 2020 + cols.index(ls))
        
        # get these columns from the result produced by process.extract
        flds = ['best_match','fuzz_ratio']
    
        # extract the best match (limit=1) at the 0th index 
        # (in order to get the values rather than a list of values)
        df_colmatch.loc[mask,flds] = process.extract(i, 
                                                     matches,
                                                     limit = 1, 
                                                     scorer = fuzz.ratio)[0]

# the output is one row per question, with the best match of that question with the list of

best-worst scaling questions¶

Some questions are asked several times with different options. These are usually referred to as best-worst scaling, and in survey research typically called MaxDiff.

It's important to be very careful with this kind of data as we can easily make wrong assumptions about how to use the results.

Typically we have a set of items and each item is compared to every other item in the set. In the case of remote vs. onsite, there are 12 statements. Each statement is compared to every other statement through a series of 6 questions that lists 6 of the items each. The respondent is asked to rate the best and worst statement in each set.

In this particular design, each statement is presented to the respondent 3 times. Because it is exactly 3 times for each statement for every respondent, the design is balanced and we can simply count the number of times a statement was ranked best and subtract from that the number of times it was ranked worst. This gives us a net score for each statement.

First, we begin by processing the "Remote vs onsite" question.
